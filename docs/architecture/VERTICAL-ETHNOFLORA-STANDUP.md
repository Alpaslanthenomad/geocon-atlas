Missing specifics in any AI-assisted draft use [EKLE:]/[ADD:] placeholders, never auto-saved as fact.

A lone Wikidata P183 stays endemism_basis='asserted' and never flips the boolean — preventing fabrication of a conservation-relevant fact.

## Inimitable angle
The moat is the RECEIPT plus the SECOND firewall, and they compound. Anyone can list medicinal plants; almost no one can hand you a public, money-blind, machine-citable Provenance Receipt that ALSO proves the ethnobotanical use-fact behind a commercial output is ABS/Nagoya-consent-gated and FAILS CLOSED without PIC/MAT. GEOCON already shipped Firewall A (commerce may only cite conservation; evidence_json_is_clean physically rejects money/PII keys at write) and the public get_chain_receipt allowlist. ETHNOFLORA adds Firewall B as a second orthogonal valve: Firewall A blocks money flowing IN, Firewall B blocks traditional-knowledge value flowing OUT without consent — a fact must pass BOTH to reach commerce. The receipt is extended (not forked) with a Part-A consent block, so the same artifact that already proves 'never money-edited' now also proves 'consent-clean', money-blind, at write/store/read. Turkey makes it sharper, not weaker: because Turkey is NOT a Nagoya Party, no IRCC can ever exist for Anatolian endemics, yet Turkish domestic PGR law still binds — so a naive 'has IRCC?' competitor would wrongly wave Anatolian material through, while ETHNOFLORA's default-deny correctly holds it ineligible. That is a correctness property a scraped list cannot replicate: a citable, dual-firewall, fail-closed consent receipt on real, provenance-labelled, harvest-pressured endemics — built on shared rails so it is a genuine second instantiation, not a clone.
