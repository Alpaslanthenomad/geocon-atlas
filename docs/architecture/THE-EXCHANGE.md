## Name
The Exchange (Venn Exchange)

## One-liner
A BEE platform-level, cross-vertical commerce home that aggregates every vertical's "ventures arm" — where investors, VENN-pioneered ventures, and industry organizations (cosmetics, clinical-study/CRO, medical and support-product firms) meet the verified value outputs of conservation work, while the conservation atlases stay pure and money never crosses into them.

## Placement
Lives at the BEE/platform root as a SIBLING of /geocon, NOT a child of it, and NOT wrapped in GeoconShell (Shell.jsx is geophyte-scoped). Route family: /exchange (with /exchange/desk as the admin back office). It gets its own minimal platform chrome, the same bare-surface pattern as app/page.js (EntryPanel) today — never the GEOCON sidebar. It is deliberately OFF the BEE vertical-picker: the Exchange is a platform service that spans all verticals, not a vertical you "enter." The existing internal deal-prep tool ("The Garden"/Bahce, VenturesRoute.jsx + VentureDetailRoute.jsx) is NOT deleted — it is relocated verbatim to /exchange/desk as the admin-only back office. The single conservation-side seam GEOCON keeps is the L5 recognition registry (/geocon/outcomes) plus the one venn_verified admin "door" (open_bridge_opportunity) in CommercializedOutcomes.jsx; the door's redirect target flips from /geocon/ventures/[id] to /exchange/desk/[id]. /geocon/ventures and /geocon/ventures/[id] become thin server redirects to /exchange (keep deep-links alive, never delete). Because the commerce DATA already physically lives in the isolated bridge schema (not in any app/geocon table), "leaving GEOCON" is UI relocation plus one additive vertical-stamp migration — not DB surgery.

## Data model
REUSE the existing isolated `bridge` schema as the cross-vertical registry — do NOT invent parallel public `bridge_*` tables (those do not exist; the real tables are bridge.opportunities, bridge.investors, bridge.opportunity_investor). One marketplace, many feeders. Additive migration only; conservation tables gain ZERO columns.

(1) VERTICAL STAMP — add `vertical_id text NOT NULL DEFAULT 'geophytes' REFERENCES public.verticals(id)` to bridge.opportunities and bridge.investors. CRITICAL: type is TEXT, matching verticals.id='geophytes' and species.vertical_id (both confirmed text) — never uuid. opportunity_investor inherits vertical via its opportunity_id FK (no column). Backfill is the default (every existing row is geophytes). The door RPC reads source vertical from species.vertical_id and stamps it; also add `vertical_id` into the frozen provenance_snapshot under source.

(2) OUTPUT REGISTRY = the frozen snapshot, generalized. bridge.opportunities.provenance_snapshot (jsonb) is ALREADY the vertical-agnostic "value output" record (today: {outcome, species, program, frozen_at}). Generalize its shape to {source:{vertical_id, kind:'commercialized_outcome', ref_id}, subject:{kind, id, name, family, iucn_status,...}, lineage:{program_id, readiness}, frozen_at}. Because subject is JSONB, not a live FK, a future vertical whose subject is a fungus/marine taxon plugs in with NO schema change. Add a `refresh_opportunity_snapshot(p_id)` admin RPC + a frozen_at display so curators can deliberately re-cite (never automatic — the freeze IS the firewall).

(3) PARTICIPANT MODEL (the two demand sides the founder named) — DO NOT create a second table; extend bridge.investors with `participant_class text NOT NULL DEFAULT 'investor' CHECK (participant_class IN ('investor','industry'))`. 'investor' = VC/angel/fund/family office (existing kind enum). 'industry' = cosmetics / clinical-study (CRO) / medical & support-product firms. Extend allowed `kind` values to add 'cosmetics','cro_clinical','medtech','supplement','nutraceutical' alongside vc/angel/family_office/corp_vc/impact_fund/grant/foundation. Add `interest_tags text[]` (WHAT value output they seek: 'extract','active_ingredient','clinical_endpoint','cultivar') so industry orgs express sourcing interest distinct from investor thesis_tags. Add `vertical_focus text[]` (empty = vertical-agnostic; populated = scoped) — this is the cross-vertical firewall analogue. ticket_min/max stay money-on-the-commerce-side-only (appetite, never deal size).

(4) ENGAGEMENT EDGE — bridge.opportunity_investor IS the interest+pipeline edge already (pipeline_stage matched→intro→nda→meeting→diligence→term_sheet→closed/passed, fit_score, notes_md, UNIQUE(opportunity_id,investor_id)). Keep it as-is; it serves investors and industry buyers identically.

(5) SOURCING BRIEFS (demand-initiated) — add a small `bridge.sourcing_briefs` table (id, participant_id FK→investors, sector, sought_output text, interest_tags text[], vertical_focus text[], notes_md, status, created_at) so an industry org can post "we need antioxidant bulb extract" and matchmaking runs buyer→listing, mirroring listing→investor.

Everything money/market/investor stays in bridge; public conservation tables stay zero-money.

## Matchmaking
Generalize the EXISTING match_investors_for_opportunity (do not replace). It is SECURITY DEFINER, admin-gated, STABLE, reads ONLY the frozen snapshot. Current score = (thesis_tag ∩ opportunity_tags)×20 + (impact_focus?25:0) + need/kind bonus (grant↔grant/foundation=20, funding↔vc/angel/impact_fund/corp_vc=15), where opportunity_tags = ['conservation','biodiversity'] + outcome.kind + lower(species.family). Three matching modes for the marketplace, all reading only snapshot + participant criteria (never a live conservation query — preserves the one-way cite):

(A) Listing→Investor (exists): keep, but surface the fit_score as a TRANSPARENT breakdown — show WHICH thesis tags overlapped (provenance is the credibility currency), not a bare number.

(B) Investor→Listings (new inverse RPC): given an investor's thesis_tags/stage_focus/ticket band, rank open listings across ALL verticals — the investor's self-serve search.

(C) IndustryOrg→Outputs (new): branch scoring on participant_class. 'industry' scores on interest_tags ∩ snapshot subject/application signals (cosmetics ↔ extract/active_ingredient + species family; CRO ↔ clinical_trial outcome kind; medtech ↔ device/material outputs). Sourcing briefs run brief→listing the same way.

Two firewall-shaped upgrades on every mode:
- VERTICAL GATE FIRST: only consider participants whose vertical_focus is empty (agnostic) OR contains the opportunity's vertical_id — so an orchid opportunity never leaks to a geophyte-scoped buyer. This is the cross-vertical analogue of the conservation firewall and must be unit-checked.
- TICKET-BAND as a HARD filter (not a soft weight): investors are not shown deals outside their appetite. ticket_min/max describe appetite, never deal value.

fit_score stays an int cached on opportunity_investor when added to the pipeline. generate_pitch_outline (reads only the frozen snapshot) is already vertical-agnostic; keep it as the listing's auto-drafted one-pager with [EKLE:] placeholders, never auto-published as fact. Add a freshness flag because the snapshot is frozen and can go stale.

Each card shows frozen provenance: italic species name, IUCN tier badge, verification-ladder badge (reuse OutcomesRoute TIERS), endorsement count, outcome kind, originating-vertical chip (emoji+brand_color from verticals — emoji is DATA, not decoration). Kanban + matchmaking mechanics unchanged. Serious tone, inline styles + var(--gx-*) tokens, no emojis in code.

## Firewall boundary
The firewall is ALREADY structural and stronger than the grounding docs imply — preserve it verbatim, widen scope by exactly one dimension (vertical_id). Verified in DB:
(1) SCHEMA ISOLATION — all commerce lives in the `bridge` schema; has_schema_privilege confirms USAGE = service_role ONLY (anon=false, authenticated=false). The anon-key client's .from('bridge...') path is unreachable; it fails closed.
(2) RPC-ONLY INGRESS — the only way in is 11 SECURITY DEFINER functions in public that first call _bridge_require_admin() (raises unless profiles.role='admin').
(3) ONE-DIRECTIONAL READ — open_bridge_opportunity is the sole bridge-creating path: it READS commercialized_outcomes/species/programs and writes a FROZEN jsonb COPY into bridge.opportunities.provenance_snapshot. It never writes back to conservation; there is no FK from conservation into bridge.
(4) ZERO MONEY IN CONSERVATION — commercialized_outcomes/commercialization_credits have no price/revenue/ticket columns; all money (ticket_min/max/currency) lives ONLY on bridge.investors.
(5) ZERO TAXONOMIC AUTHORITY IN COMMERCE — bridge references species only as a denormalized snapshot, never as source of truth.

To extend cross-vertical WITHOUT weakening: add `vertical_id text` to bridge tables (commerce now knows which vertical it cites, still one-directional); the door reads the source vertical from species.vertical_id (commerce still never imports market/investor data INTO conservation). The /geocon/ventures redirect keeps the firewall intact because the door logic is unchanged. CITE-ONLY, READ-ONLY, ONE-DIRECTIONAL holds verbatim. HARD RULES for the build: never grant bridge USAGE to anon/authenticated; never enable-RLS-as-the-gate as a substitute for schema isolation; the new BEE participant role must be fully DISJOINT from conservation roles and can never reach the conservation interior (species localities, experimental results); industry-org PII (contact_email) stays admin-gated behind _bridge_require_admin and never on a public route; ticket bands describe appetite, not deal size; no deal value/pricing/revenue ever appears on a listing.

## Firewall critic verdict
- firewall_ok: **False**
- verdict: REJECT as written. The structural firewall it builds on is real and the design's core moves are sound: schema isolation is confirmed (bridge USAGE = service_role only; anon/authenticated false), the door RPC open_bridge_opportunity is genuinely one-directional (reads conservation, writes a frozen JSONB copy into bridge, never writes back, no FK from conservation into bridge), commercialized_outcomes/credits carry zero money columns, the vertical_id stamp is type-correct (verticals.id and species.vertical_id are both TEXT), and the UI relocation + redirect + nav-target flip touches no conservation table. The vertical_id extension widens scope without adding a money path. BUT two real problems block approval: (1) the design asserts a clean 'zero money in conservation' baseline that the live DB contradicts — anon-readable public tables market_intelligence, species (market_area/market_size/spinoff_link), species_stories, and commercial already hold money/market columns, so the P0 firewall contract is built on a false invariant; and (2) the optional 'Listed on the Exchange' cite-back is specified as reusing 'an existing bridge RPC' when every bridge read RPC is admin-gated and the outcome card serves non-admins — making the only buildable versions of that feature firewall violations (a non-admin GEOCON read of commerce data, or exchange state written into a conservation row). Fix the baseline claim and convert the cite-back from 'defer' to 'no-build as specified,' and the rest of the design is firewall-safe to proceed phase by phase.
- leaks:
  - FALSE BASELINE / EXISTING MONEY IN ANON-READABLE CONSERVATION TABLES: The design's load-bearing claim 'ZERO MONEY IN CONSERVATION ... Verified in DB' is false. Live DB shows money/market columns sitting in public, RLS-on, anon-SELECT=true tables that are NOT in the bridge schema: public.market_intelligence (market_size_usd, price_range, market_cagr, revenue_model, market_readiness), public.species (market_area, market_size, spinoff_link), public.species_stories (market_narrative), public.commercial (market_type). These are pre-existing, but the design codifies the false invariant as the P0 'firewall contract you regression-test every later phase against' (build_phases P0) and repeats it verbatim in firewall_boundary item (4). A regression test asserting 'zero money columns in public' will either fail immediately or be written to ignore these tables, hiding the real exposure.
  - INCOHERENT / LEAKY 'LISTED ON THE EXCHANGE' READ-BACK CITE: geocon_extraction item 5 and open_questions both propose an OPTIONAL one-way read-only cite on the venn_verified outcome card, 'computed by an existing bridge RPC.' Verified: ALL bridge read RPCs (list_bridge_opportunities, get_bridge_opportunity, list_bridge_investors, list_opportunity_pipeline) call _bridge_require_admin() and raise for non-admins. The outcome card in CommercializedOutcomes.jsx renders for logged-out and non-admin users. Therefore NO existing RPC can supply this cite to the audience that sees the card. Building it as specified forces one of two firewall violations: (a) a NEW non-admin/anon RPC that reads the bridge (commerce) schema and returns it into a conservation surface = a GEOCON read pulling commercial data; or (b) persisting 'is-listed' / listing-id state into a conservation table = market/exchange state landing in a conservation record. The design mislabels this as a safe reuse of an existing read; the stated implementation path is a leak.
- required fixes:
  - Drop the false 'zero money in conservation' invariant from P0/firewall_boundary. Either (a) scope the firewall assertion precisely to the tables it is actually true for (commercialized_outcomes, commercialization_credits have no price/revenue/ticket columns) and explicitly EXCLUDE the legacy market tables, OR (b) open a separate remediation track for market_intelligence / species.market_* / species_stories.market_narrative / commercial.market_type before claiming a clean baseline. The P0 regression test must enumerate exactly which public columns are allowed to be money/market and fail on any NEW one, rather than asserting a blanket zero that is already violated.
  - FORBID the 'Listed on the Exchange' read-back cite as currently specified. Do NOT build it via a new non-admin bridge-reading RPC and do NOT persist exchange/listing state into any conservation table. If a one-way 'this work yielded a listing' signal is ever wanted, it must be a derived count produced inside the bridge boundary and surfaced ONLY on the /exchange (BEE-root, non-conservation) surfaces, never injected back into a /geocon outcome card. Per the design's own 'Lean: omit at launch' — make that a hard NO-BUILD, not a deferral, and strike the 'computed by an existing bridge RPC' language since no such RPC exists for non-admin callers.
  - In the matchmaking generalization (P1/P2/matchmaking), keep every new/extended RPC (match_investors_for_opportunity inverse + IndustryOrg mode + brief->listing) reading ONLY bridge.opportunities.provenance_snapshot (the frozen JSONB) and bridge participant criteria. Add an explicit unit assertion that none of the new RPCs issue a live SELECT against public conservation tables (species, programs, localities, market_intelligence), so 'reads only the frozen snapshot' is enforced, not just stated. The existing match_investors_for_opportunity already does this correctly; preserve that pattern.
  - Preserve the existing ingress posture verbatim and do not weaken it: the bridge RPCs are EXECUTE-granted to 'authenticated' and fail closed only via _bridge_require_admin(); the new BEE participant role must be added WITHOUT granting any bridge RPC to it and WITHOUT granting bridge USAGE to anon/authenticated. Industry-org contact_email and all participant PII must stay behind _bridge_require_admin and never reach a public /exchange route.
