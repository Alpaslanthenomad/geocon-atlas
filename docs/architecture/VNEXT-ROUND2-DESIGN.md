# v-next Round 2 — design (program lifecycle + rights + tics + first environment)

> 5-agent, on the locked frame. Design only -- no build. Almost zero new schema (the engine maps onto existing free-text status + program_members columns; only a thin positions table is new).

## The round-2 spec (one spine)
ONE SPINE, ONE VOCABULARY. GEOCON v-next is THE BOOK: a researcher's home is a PORTFOLIO of POSITIONS, and a POSITION is the single atom from which everything else is a view.

THE ATOM. A POSITION = (researcher x species x work-area axis), where axis is X=safeguard, Y=knowledge, Z=value. It is one thin new row in the only genuinely-new table, `positions(researcher, species, axis, status, program_id?)`. Everything else in the spine is already shipped and is reused as a LENS over this atom — no second model anywhere.

THE VERB (unchanged, the one shipped engine). Evidence enters a position ONLY by asserting a DOI-backed fact, which mints a money-blind Provenance Receipt (band 0.00-1.00). Receipts attach to the SPECIES (chain_link_fact = species x chain-node), never to the program. This single structural fact — receipts live on the species, not the program — is what makes three of the four findings collapse to almost nothing (handoff is rights-neutral, lineage is computed for free, the species page outlives everyone). Conviction is a separate subjective dial, never credited, never on a receipt.

THE THREE STATES OF AN ATOM. (1) Private position: a researcher quietly accumulating receipts on their own portfolio. (2) PROMOTED -> a PROGRAM: a public, joinable LIVING VALUE ENGINE, never "done". (3) The species page: the durable AGGREGATOR where all programs (active + passive) and all receipts roll up; it outlives any researcher because the receipts were never the program's.

THE PROGRAM IS A LIVING VALUE ENGINE. Status is free `text` on the shipped `programs` table (NOT an enum), so the lifecycle is config, not migration. Four values, NO terminal "done": Draft (pre-public), active (worked now), passive (dormant = a STANDING OPEN CALL, the gap rendered as an invitation), revival (one-call transient on handoff). Auto-demote active->passive by ONE number (no completed tic in 180 days) via the pg_cron already running. Promote/revive flips back inside the existing complete_program_tic / respond_to_join_request calls. A passive program publicly advertises its locked terminals and unfilled gates — "open work here, pick it up" — and a new researcher revives it via the SAME join handshake plus one is_solo_carrier flag and one grace-window branch. "What one could not, another can" is literally the passive->active transition under a new carrier; the first researcher's receipts stay credited because they are on the species.

THE TIC-ARCHITECTURE (the body of every program). ONE template work-graph authored as ROWS in the shipped `tic_definitions`, over the Venn region engine (X/Y/Z) and stages (foundation->...->governance). A position is a CURSOR onto it, not 47k graphs. Required vs optional vs conditional is the R1 principle: REQUIRED only when a downstream output is scientifically invalid/unsafe without it (a validity gate), OPTIONAL when it only raises strength, CONDITIONAL when required only if a specific output is targeted. X (safeguard) is gate-heavy because the science is unforgiving (you cannot bank seed of unknown storage behaviour; you cannot reintroduce without genetic breadth + threat-abatement + a secured site + >=4yr monitoring). Y (knowledge) is deliberately FLAT/enrichment-heavy with only two hard gates (taxonomy root + data-dependency); a "parked/exploring" Y is first-class and Published-Knowledge is a terminal win of equal standing — Y exit is NOT mandatory. Z (value) is a short required trunk (voucher+ABS -> extract -> metabolite-profiling = THE FAN TRIGGER) that fans into hundreds of TIER-LABELLED money-blind potentials, each a point in (activity-class x value-domain x indication) at an evidence tier T0-T4 capping strength. The fan is a JOIN over ChEMBL/PubMed gated by what the researcher actually evidenced on THIS species — never asserted of the species. Grounded live: galantamine->AChE(CHEMBL220, max_phase 4)->Alzheimer; colchicine->tubulin->gout(T4)/CV(Phase2-4)/oncology(T1) off one mechanism node.

THE RIGHTS MODEL (out of GEOCON, by construction). Rights/benefit-share flow through the receipt/evidence LINEAGE, not the program. Because receipts are species-scoped, "trace across parallel programs" is the default shape of the data, not a join to build. GEOCON emits ONE read-only RPC (`get_potential_lineage`) — an allowlist projection, money-blind like the shipped get_chain_receipt — returning the weighted evidence closure (load-bearing/enriching/foundational, weight = fill_strength x necessity-multiplier) plus the ABS/Nagoya block. The Exchange (money side) multiplies that frozen, one-directional snapshot by money it alone holds: ABS off the top to the source community, then a weighted pool ratified by humans. The receipt is your share certificate — money-blind citation in GEOCON, fair-share basis in the Exchange.

THE ENVIRONMENTS (one work-area bench at a time). Each axis gets a tailored bench generalizing the shipped Thesis Workbench (tabbed surface + pure-JS compute pane + DOI evidence binder + mint-to-receipt). FIRST = the IUCN Red List Assessment bench (X/Y), because it is ~70% already shipped and merely orphaned.

THE ONE SENTENCE: a researcher declares interest in a plant-and-area; a living engine appears made of the gap; collaborators are invited; work mints receipts to the species; if the researcher drifts the engine becomes a lit signpost another picks up; the species page quietly accumulates everything and outlasts them all — nothing is ever finished, because a living value engine, like the science it tracks, never is.

## Program lifecycle (living value engine)
FINALIZED, SIMPLE. The lifecycle is STRING VALUES + transition guards in RPCs that already ship — ZERO new tables (the one new table, `positions`, belongs to the atom/reconception, not the state-machine).

STATES (programs.status, free text, default 'Draft' — NOT an enum, deliberately, so states are config not migration):
- Draft (exists): position just promoted, owner still framing; owner-only.
- active: a living engine with a working carrier, producing tics/receipts now; public face + member interior.
- passive: dormant = a STANDING OPEN CALL, flagged "open for revival/handoff"; the gap (unfilled gates, locked terminals) IS the public body.
- revival: one-call transient doorway during a handoff handshake; flips straight back to active.
- (withdrawn): rare, reversible "retract this listing" for a duplicate/mistake — NOT "done"; routes never delete (constraint #4).
There is deliberately NO completed/closed/archived. "Done" is a category error for a value engine; the closest thing to closure is passive, and passive is defined as an INVITATION.

TRANSITIONS (all on shipped RPCs):
- active->passive AUTO: one number — no program_tic reached completed in 180 days — swept nightly by the pg_cron already in the project. No per-tic decay model.
- active->passive MANUAL: owner clicks "Park" -> one UPDATE status='passive', next_action='Open for revival'.
- passive->active: any completed tic by a member, OR a successful handoff — flips back INSIDE the existing complete_program_tic / respond_to_join_request call. No separate reactivate RPC.

PROMOTION (position->program): one thin wrapper RPC `promote_position_to_program` that calls the shipped `create_program` pre-filled from the position (axis->program_type map X->Conservation, Y->Research, Z->Commercial, mixed->Hybrid — values that already exist), sets positions.program_id, and sets status='active' directly. NO evidence copied — the program renders the species' existing receipts through the axis cursor (the rights-lineage win, for free).

DECLARE-INTENT AUTO-FORMATION: one orchestrator RPC `declare_intent(species, axis, message)` composing existing parts — insert the position, auto-promote, then call the shipped `suggest_collaborators` for top-k matched + 1-2 random-discovery slots and write `invited` rows into program_members (columns invited_at/invited_by/invitation_message already exist). No new matcher, no invitation table, no notification system.

HANDOFF (the heart): the SAME request_to_join_program + respond_to_join_request, with ONE added branch — if the program is passive and no owner responded within a 30-day grace window, an accredited steward (or system policy) may approve; on approval the reviver becomes active with is_solo_carrier=true (column exists), the prior owner is DEMOTED to member (never deleted — attribution + lineage intact). The ONLY genuinely-new logic in the whole lifecycle.

MIXED-AREA TEAMS: emergent, NOT a program field. Each member's axis lives in the shipped program_members.pathway_scope (jsonb); the program's areas = the distinct set of members' axes; the shipped get_program_region_status already aggregates X/Y/Z + integrated_core. The Venn engine was built for exactly this. Add nothing.

MULTIPLE PROGRAMS PER PLANT: already allowed (programs.species_id non-unique). No merge, no canonical "the program," no referee — parallelism is the truth; the species page is the join.

NET NEW: one table (positions, the atom), three thin composition RPCs (promote_position_to_program, declare_intent, park_program), two added branches in shipped RPCs (passive-reactivate; grace-window handoff), one pg_cron sweep, one SELECT-list addition (status + is_open_call to get_programs_for_species_rich). No enum, no membership/invitation/handoff/matchmaker table, no evidence on programs, no money column.

## Rights model (receipt lineage)
FINALIZED, SIMPLE — the cheapest piece in the arc (~85% ships). The structural insight, discovered not designed: chain_receipt -> chain_link_fact (species x node), with NO program_id. So rights flow through the receipt/evidence LINEAGE across whatever parallel programs touched the work, and "trace across programs" is the DEFAULT shape of the data, not a join to build. A per-program cap-table would have been the harder, worse system.

THE UNIT: the receipt ("the receipt is your share certificate"). A program is just a workspace that produced some receipts; it has no claim of its own.

THE LINEAGE (computed, never stored): a Z value-potential's evidence closure = two walks over already-shipped tables, no new graph table. Walk A (vertical): from the potential's anchor chain node, follow chain_link_type.path (ltree) + the R1 requires-backbone toward the root; every filled fact on that required path and its receipts are in the closure. Walk B (temporal): for each fact follow chain_evidence.supersedes_id to the head; only the current non-retracted head carries weight, superseded rows are cited (foundational), never double-paid. A bounded recursive CTE; species-local. Net-new GEOCON persistence: ZERO.

THE WEIGHTING (the only judgemental part, and the substrate already encodes the judgement so we invent no scoring model): R1's REQUIRED-vs-OPTIONAL split IS the weighting. weight(receipt) = fill_strength x necessity-multiplier, where multiplier = 1.0 for is_gate/requires (load-bearing/equity), 0.5 for enriches/support (equity, lower), 0.0 for off-path/foundational/superseded (cited, reputational only, NO equity). A contributor's raw share = sum of their receipts' weights / total closure weight — the same weakest-link math recalculate_evidence_strength already runs. This credits the ACT of proving a load-bearing fact, weighted by strength and necessity, so a parallel-program chemist whose assay is load-bearing is included automatically and the genus-taxonomy verifier is not toll-gating every future potential.

HONEST EDGE CASES (integrity, not exceptions): foundational/generic = cited not equity (protects the commons of basic facts); superseded = the head is equity, the prior is cited; enriching-only = equity at 0.5; tier honesty rides along (a T1 potential has a thin closure — correctly — and the split deepens as enrichment tics land). THE ABS/NAGOYA SLICE is carved FIRST, off the top, from the shipped bridge.abs_consent (source_community, is_traditional_knowledge, consent status); abs_consent_is_commerce_eligible must pass or the opportunity cannot open — the community is paid before any researcher; TK-derived potentials carry a heavier reserved share. The biopiracy guard made financial.

THE FIREWALL SPLIT (the load-bearing design choice): GEOCON side emits the LINEAGE, never a split — ONE new read-only RPC `get_potential_lineage(p_outcome_id)`, modelled exactly on get_chain_receipt's allowlist discipline (SECURITY DEFINER, set search_path=public, granted to authenticated), returning {receipt_pid, node_label, contributor, fill_strength, role in {load_bearing|enriching|foundational}, supersedes_pid?} + the ABS block, with NO currency and NO percentage — it physically cannot carry a number with a currency, and evidence_json_is_clean already guarantees nothing money-shaped entered the substrate. The EXCHANGE side (money) freezes that snapshot one-directionally into the shipped bridge.opportunities.provenance_snapshot at open time and does ABS-off-the-top -> weighted-pool -> human ratification. GEOCON never sees the pool, the deal, or the dollars. One object (receipt), two roles (citation in GEOCON, share-basis in the Exchange), multiplication only where money already lives.

THE QUIET HIGH-VALUE GRAFT: declare_commercialized_outcome already accepts p_initial_credits and writes commercialization_credits — today hand-typed. The lineage layer PROPOSES that credit list from the evidence closure, turning a hand-maintained guess into a derived, defensible, receipt-backed list the declarer ratifies.

NET NEW IN GEOCON: exactly ONE RPC (get_potential_lineage) + one "Rests on" UX panel + one line on /receipt/[pid]. The money math lives only in the Exchange. FOUNDER CALL: the ABS reserved fraction is a policy number (lives in the Exchange), and whether foundational work earns a token slice (recommend reputational-only) is a one-line multiplier-floor change.

## Tic-architecture (deepened)
FINALIZED AT THE DESIGN LEVEL. ONE template work-graph = ROWS in the shipped tic_definitions (PK id text, convention <wheel>.<snake>; fields stage / default_region / priority / is_required / evidence_required / gate_validation_rule). 12 active tics anchored verbatim across foundation(3) + field_lab(9); the empty propagation/deep_work/deployment/governance stages are exactly where the new X/Y/Z rows go. A position is a CURSOR; routing separates AVAILABILITY (binary: prereqs+gate+firewall+min-strength) from STRENGTH (the rendered 0-1). Five R1 primitives, two genuinely new + thin: a tic->chain weld + a thin tic_edge table. Required = output invalid/unsafe without it; Optional = raises strength; Conditional = required only if a specific output is targeted.

SHARED TRUNK (ships): sci.taxonomy_verified [R, ROOT — a fact about a misidentified plant is fabricated], sci.specimen_documented [R, the voucher + ABS provenance stamp], sci.morphological_characterization [R], cons.threat_analysis [R], cons.baseline_assessment [R]. After the trunk the axis selects which firewall_class of terminal the unlock rule admits (X/Y -> conservation_only; Z -> translational).

X — SAFEGUARD (gate-heavy by necessity): X.1 Assessment->Red List is the LOWEST-barrier win (cons.redlist_assessment terminal conservation.assessment.redlist_filed, deps = trunk+threat+baseline, all shipped) — bridges the 47k stub gap. X.2 Ex-situ pivots on the ROUTER cons.seed_storage_behaviour (orthodox->seed_banking terminal; recalcitrant/intermediate->invitro->cryo_protocol FORCED terminal). X.3 Propagation = the reintroduction precondition stack (germination_protocol, propagation_protocol [OR-group fill_rule=max_across_siblings], genetic_diversity_captured). X.4 Reintroduction is the heaviest terminal: translocation_executed opens conservation.reintroduction PROVISIONAL; post_translocation_monitoring (>=4yr) CLOSES it (the single most-cited failure mode is declaring success before monitoring). habitat_enrichment is a lighter standalone win.

Y — KNOWLEDGE (deliberately FLAT): only two hard gates — taxonomy root + data-dependency (cannot publish before the data; cannot validate a structure before isolating it). Everything else ADDITIVE, each filling a distinct node and raising strength (phenology, ecological_niche, genetic_characterization [also feeds X breadth], pollination_biology, cytogenetics, phylogenetic_placement, population_genetics). Terminal sci.peer_publication -> knowledge.published, a real win of EQUAL standing. Parked/exploring is first-class; Y exit is NOT coerced.

Z — VALUE (short required trunk -> wide tier-labelled fan): TRUNK [R] = specimen_documented+ABS (an is_gate — missing ABS multiplies every Z cap toward 0) -> chem.extract_prepared -> sci.metabolite_profiling [R, THE FAN TRIGGER: the instant a compound is named, every external ChEMBL/PubMed edge flips AVAILABLE @ T1 at once, populating the species_compound weld]. ENRICHMENT [O] promotes INDIVIDUAL points up the tier ladder without gating visibility: chem.crude_screen (T1->T2 for that point), chem.fractionation, chem.compound_isolated, chem.pure_potency (T2->T3), chem.mechanism_confirmed (T3->T4). EXIT chem.value_potential_declared [C] -> translational.* ladder (self->peer->org->venn_verified), a money-blind POTENTIAL never a product; only at venn_verified may the Exchange cite it, one-directional.

THE Z VALUE-POTENTIAL TAXONOMY (the fan terminals, authored once, lit per species as a JOIN over ChEMBL/PubMed — never asserted of the species): a point = (ACTIVITY CLASS x VALUE DOMAIN x INDICATION) at an EVIDENCE TIER. Activity classes (~12, each a real ChEMBL anchor: enzyme-inhibition/cholinergic = galantamine CHEMBL659->AChE CHEMBL220 IC50 100nM max_phase 4; cytoskeletal/antimitotic = colchicine CHEMBL107->tubulin; antiproliferative/antiviral/anti-inflammatory = lycorine; etc.). Domains (~7 = the translational terminal type: pharma/nutra/cosmeceutical/agro/vet/industrial/reagent). Tiers (the HONESTY axis = a strength CAP, same line as no-fabrication): T0 genus-inferred ~0.15, T1 literature-on-compound/this-accession-untested ~0.40, T2 this-species-extract-screened ~0.65, T3 pure-from-this-accession-assayed ~0.85, T4 mechanism/clinical-precedent ~1.0. Worked Galanthus elwesii: profiling names galantamine+lycorine -> the fan fires every edge @ T1 at once (galantamine->AChE->Alzheimer flagged "molecule verified, this accession untested"); enrichment then promotes single points; colchicine->tubulin proves one mechanism spawns differently-mature potentials (gout T4 / CV Phase2-4 / oncology T1-2).

THE TWO GENUINELY-NEW PRIMITIVES (thin): (1) species_compound weld (species x molecule + ChEMBL_id/InChIKey + provenance) populated by metabolite_profiling — Z-ONLY. (2) tic_edge (requires/enriches) authoring the necessity backbone — degrades gracefully to "on the ltree required path vs off it" until authored. NOTE: X benches need NEITHER (their dependency set is already-shipped trunk tics evaluated by the existing gate). FOUNDER CALLS surfaced: tier floors (visible always / available >=T2 / exit >=T3); how wide T1 broadcasts publicly (recommend gating T1 to members via chain_link_fact.sensitivity); is X reintroduction gated (recommended) and is provisional a displayed win; the Z external-edge confidence floor (>=1 curated ChEMBL mechanism OR >=2 independent PubMed assays — the unresolved lycorine ChEMBL-ID this session is exactly why this floor matters); who authors the tic_edge welds.

## First work-area environment
THE IUCN RED LIST ASSESSMENT BENCH (an X/Y conservation position) — wins decisively on value x simplicity x alignment.

WHY FIRST. VALUE: it attacks the platform's single biggest, most-cited gap — 44,146 of 47,413 species (93%) are Not Evaluated; only 3,267 have a real status. A filed Red List assessment is a REAL conservation action — an X/Y output where conservation IS the value — so it needs NO Exchange, NO market, NO network to be worth doing. For a fundless solo researcher at cold start it is the rare bench whose output is a finished, citable, real-world-useful artifact on day one, and it moves the North-Star (a fact from 0) at the largest available scale. SIMPLICITY: it is ~70% already shipped and merely ORPHANED — IucnAssessmentEditor.jsx exists end-to-end (category CR/EN/VU/...; the real A/B/C/D/E criteria checklist incl. B1ab(iii)/C2a(i)/D1/D2; 5 narrative sections; a draft->peer_review->submitted->published state machine; SIS .json export at /api/v1/iucn/<id> — all verified this session in the file). The iucn_assessments table exists. The chain terminal exists (conservation.redlist.assessment_workflow, firewall_class=conservation_only). The trunk tics exist (sci.taxonomy_verified, cons.threat_analysis, cons.baseline_assessment — the exact {taxonomy, threat, baseline} dependency set R1 named). The ONLY real defect: advance_iucn_assessment flips a status string and mints NOTHING. ALIGNMENT: a pure X/Y position, so the firewall is trivially safe (conservation_only terminal, no translational fan, no money surface) and it needs NONE of the not-yet-built Z machinery (no species_compound weld, no tic_edge, no external joins). It is the cleanest possible proof that a work-area environment generalizes the Thesis Workbench, because the Workbench's own primitives map one-to-one.

THE BENCH (Thesis Workbench generalized; tabs): (1) ASSESSMENT EDITOR — reuse the shipped IucnAssessmentEditor as-is. (2) PARAMETER CALCULATOR — NEW, thin, pure-JS (mirrors AnalysisPane.jsx): AOO/EOO from locality points (2x2km grid / minimum convex polygon), generation length, population-reduction % over 3 generations/10yr (Criterion A), small-population thresholds (C/D); each computation a reproducible run so criteria stop being unsupported chips. Locality inputs honour the shipped sensitivity tiering (the verb already rejects raw coordinates as PII). (3) EVIDENCE BINDER — reuse the ReferenceLibrary/resolve-doi+CrossRef path so each section cites a GEOCON receipt or an external paper. (4) CRITERIA HELPER — the existing checklist, lightly enriched with plain-language v3.1 thresholds; AI rationale uses [EKLE:] and is NEVER auto-saved as fact.

THE ONE NEW WIRE (verb weld): on publish, advance_iucn_assessment ADDITIONALLY calls the SHIPPED assert_fact_and_mint(species, link_type=>conservation.redlist.assessment_workflow, value=>{proposed_category, criteria[], assessment_id, status:published}, doi=>assessment/SIS PID) — writing chain_link_fact + chain_evidence (guarded by evidence_json_is_clean/_public_text_is_clean) and minting a money-blind receipt at /receipt/[pid]. The same publish completes the conservation tic via complete_program_tic on a promoted position (advancing the X/Y gate), or simply lifts an Active position's evidence band off 0 and posts to the Tape. DATA TABLES: ZERO new substrate in v1 (reuse iucn_assessments + chain_link_fact/evidence/receipt). OPTIONAL one thin append-only reproducibility table iucn_param_run(id, assessment_id FK, method, inputs_json, result_json, created_by, created_at) — RLS mirroring iucn_assessments, parallels thesis_analysis_run exactly — and only if the calculator ships in v1.

FELT EXPERIENCE: a solo researcher opens a Galanthus x Safeguard position, sees a calm checklist of what a Red List filing needs (taxonomy verified? threats analysed? AOO computed?), fills the gaps the bench makes obvious, computes AOO/EOO from real points, writes 5 short cited sections, hits Publish — and watches the position band lift, a citable receipt mint, the Tape record "Galanthus elwesii -> VU assessment filed," and the species page gain a real status where there was none. The gap was the product; the bench closed it.

## Complexity stress-test (the v1 lesson)
HONEST CRITIQUE AGAINST THE FOUNDER SIMPLICITY MANDATE (the v1 arc died of complexity + design + tech, NOT the concept). I ran the full design through the bench-arc-resemblance check and cut where complexity crept.

WHAT THE WHOLE SPINE ADDS, TOTAL: ONE new table (positions). FIVE new thin RPCs (promote_position_to_program, declare_intent, park_program, get_potential_lineage, plus the iucn publish-weld which is an EDIT to an existing RPC not a new one). TWO branches in shipped RPCs. ONE pg_cron line. ONE SELECT-list addition. TWO Z-only primitives DEFERRED out of the first environment entirely (species_compound, tic_edge). That is the entire net-new surface for a four-part reconception — because receipts-on-species, free-text-status, the Venn region engine, and the IUCN editor were all already shipped.

WHERE COMPLEXITY CREPT, AND THE CUT:
1. Program lifecycle wanted an enum + completed/closed + a decay model + handoff/invitation/matchmaker tables. CUT to free-text status (config not migration), NO terminal state (the model forbids it), ONE 180-day number swept by the existing cron, and REUSE of the join handshake + is_solo_carrier for handoff. Net-new tables from the lifecycle: ZERO.
2. Mixed-area teams invited a programs.axes column + a "convert to mixed" action + a team-composition table. CUT — mixed-ness is an EMERGENT read over program_members.pathway_scope + the shipped get_program_region_status. Add nothing.
3. The rights model invited a per-program cap-table engine + a share ledger + benefit-share fields on programs/members + a money column. CUT to ONE read-only allowlist RPC in GEOCON; the money math lives ONLY in the Exchange. The cut was free because receipts were never program-owned.
4. The Z fan invited authoring potentials per-species (47k x hundreds of rows) and a from-scratch external-DB integration. CUT to ONE template authored once + a JOIN lit per species, and the whole Z environment is DEFERRED behind the conservation bench.
5. The first environment invited a Pyodide/WASM stats kernel, GBIF live-ingestion, multi-assessor real-time co-editing, and an official SIS API push. ALL CUT from v1 — pure-JS arithmetic only, pasted locality points, single-player, static SIS export. Even the reproducibility table is optional.

BENCH-ARC-RESEMBLANCE CHECK (does this look like the reverted personalization/bench arc?). The reverted arc failed for three reasons; this design fails NONE of them:
- It was SHALLOW (a station chip, persona axes, cosmetic) — this design's atom is the SHIPPED verb (assert->receipt), the one thing with real depth; every surface is a lens over real evidence, not a decoration.
- It was TOO RADICAL (3-axis personas, claim-species/chain-heal as new mental models) — this design introduces exactly ONE new noun (position) and otherwise REUSES nouns the founder already approved (program, receipt, species, the Venn X/Y/Z that already ships).
- It was a COMPLEXITY+TECH failure (a big new subsystem) — this design's net-new is one thin table + a handful of composition RPCs; three of the four findings collapse to almost nothing precisely because the substrate already enforces the model.
REMAINING RESEMBLANCE RISK (named honestly): the WORD "bench" and a tabbed personal surface are exactly what was reverted. MITIGATION: the first bench is the IUCN environment that is ~70% ALREADY BUILT and shipped (not a from-scratch personal hub), it delivers a real-world artifact (a filed assessment) rather than a personalization feel, and it is built ONE at a time. If even this reads as "bench arc redux" to the founder, the fallback is to ship the verb-weld ALONE (publish->receipt) with NO new tabs — proving the spine with the smallest possible surface before any bench framing. The single biggest standing complexity risk is tic_edge authoring (the necessity backbone) — explicitly DEFERRED and degraded-gracefully so it cannot block the first build.
