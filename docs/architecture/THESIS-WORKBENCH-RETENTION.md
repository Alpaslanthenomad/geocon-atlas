# GEOCON — Thesis Workbench + Researcher Retention (design)

> 5-agent workflow: map current + sticky-platform research + workbench/retention design + synthesis. Build-first next.

## Thesis workbench vision
The GEOCON Thesis Workbench is the surface where a geophyte researcher does the entire empirical chapter -- data, analysis, citations, figures, prose, export -- without ever opening R, SPSS, Zotero, or Overleaf, because every artifact produced inside it is bound to the atlas species/program it came from, timestamped as a tic, and mintable into the SAME money-blind, citable Provenance Receipt that already powers /receipt/[pid]. It is not "Overleaf for plants." It is the only writing-and-analysis environment where the act of doing the work simultaneously (a) cites the atlas's verified facts, (b) credits you on your public Researcher Passport, and (c) emits a tamper-evident, reproducible, provably-money-blind record a journal or supervisor can verify. Architecturally it EXTENDS the existing thesis_tracks substrate (RLS: student + supervisor) rather than spawning a new silo: a thesis gains a per-thesis reference library, a dataset+analysis pane that runs real statistics in-browser, recharts figures bound to analysis runs, a chaptered markdown writing surface that cites the library and embeds the figures, and a mint-receipt path. Each surface references the others -- the writing cites the library; the library can cite atlas receipts; figures embed analysis runs; analysis reads hashed datasets -- and that cross-linking, not any single feature, is the gravity (the Notion/Overleaf lesson). Every load-bearing primitive is already on-stack and verified in package.json: xlsx 0.18.5 (dataset import is essentially free), recharts 3.8.1 (figures), react-markdown 10.1.0 + rehype-sanitize 6.0.0 (safe render), @anthropic-ai/sdk (the same drafting path Grant Studio uses), Supabase RLS+RPC (storage+gating), and get_chain_receipt (the mint target). The new libraries are small and additive (citation-js, simple-statistics, jStat -- all pure JS, no WASM on the critical path).

## Retention north star
Keep researchers in GEOCON by making it the place where their work becomes CITABLE, CREDITED, and PROVABLY MONEY-BLIND -- so leaving forfeits accumulated, interlinked, externally-credentialed state (a reference library, a live thesis, hashed analysis runs, minted receipts) that no competitor can reissue. Retention is never the feature; it is the gravity of the state the feature produces plus the people and reputation bound to it. This gravity is OPEN, not captive: everything exports (BibTeX/CSL-JSON, CSV, LaTeX/DOCX/MD) -- the researcher stays not because they cannot leave but because leaving costs the citations-to-the-atlas, the reproducibility proof, the supervisor's shared state, and the public credit, none of which an external toolchain can reproduce. The single governing metric remains the existing North-Star one: an evidenced fact moved from 0. Every retention mechanism must increase the count or depth of evidenced facts (a closed gap, a verified analysis, a credited contribution) or it is theatre. The one durable lock-in is the money-blind, citable Provenance Receipt -- reputational, not coercive: a researcher cannot abandon the CV entries, ORCID links, and paper citations that point at receipts only GEOCON can issue and only GEOCON's Postgres-enforced firewall can vouch for.

## Inimitable angle
The workbench is inimitable because it is welded to the citable-receipt + credit + firewall moat at three points no generic toolbox can copy. (1) WORK BECOMES A RECEIPT: a finished analysis run or defended chapter mints into the SAME Provenance Receipt object that already powers /receipt/[pid] -- a persistent-identifier, money-blind, CrossRef-resolved, citable artifact. The receipt machinery, the firewall CHECK (evidence_json_is_clean), and the citation_string already exist; the workbench just generalizes the receipt's source from chain_evidence to analysis_run / thesis_chapter via an allowlist projection. So doing the thesis work IS producing citable scientific records -- Overleaf produces a manuscript with no provenance, SPSS produces numbers with no record, the workbench produces a tamper-evident, reproducible, citable object. (2) WORK BECOMES CREDIT: every minted receipt feeds the credit loop and surfaces on the public Researcher Passport (/r/<handle>) built this session -- attributable, public, verifiable credit for analysis and writing done inside the system, which a generic Overleaf clone (a bare PDF) cannot give. (3) WORK IS PROVABLY UNTAINTED: the conservation/commerce firewall is WRITE-ENFORCED in Postgres -- stats run money-blind on a conservation-data projection and any persisted output passes the money/PII-blind CHECK, so outputs are not just citable and credited but provably free of commercial contamination, a credential no commercial research tool can offer because none has a structural firewall. The triple -- reproducible (the OSF lesson) + credited (the Passport/credit loop) + provably money-blind (the firewall) -- exists in NO incumbent (Overleaf, Benchling, Zotero, SPSS, OSF each offer at most one). And because the gravity is open (everything exports), the moat is reputational and structural, not coercive -- the durable kind: the researcher gains a credential they want to keep rather than being held hostage by un-exportable data, the explicit anti-Benchling stance.

## Statistical-analysis tool spec
CONCRETE BUILDABLE v1 -- the "Analysis" pane inside a thesis chapter (route: /geocon/thesis/[id], new Analysis tab; component AnalysisPane.jsx).

INPUTS: a thesis_dataset (imported via xlsx 0.18.5 -- already in deps -- from CSV/XLSX, or a query against atlas species data already in Supabase). The user picks columns and a test from a typed menu. Each dataset carries a sha-256 content hash computed at import (the reproducibility anchor).

METHODS in v1 (pure JS, NO WASM, so zero load cost):
- Descriptives: n, mean, median, SD, variance, min/max, quartiles -> simple-statistics.
- One- and two-sample t-test (1/2-sided), with CI and p-value -> jStat (jStat.ttest / studentt).
- One-way ANOVA (F-test) + Tukey HSD post-hoc -> jStat (anovafscore / tukeyhsd).
- Pearson and Spearman correlation -> simple-statistics (sampleCorrelation) + rank transform for Spearman.
- Simple linear regression (slope, intercept, R^2, residuals) -> simple-statistics (linearRegression / linearRegressionLine).
These five cover the bulk of an empirical geophyte thesis (e.g., does bulb mass differ between two populations; does altitude predict alkaloid concentration; does treatment affect germination rate). simple-statistics has no deps and runs in all browsers; jStat is tiny. Gaps in jStat (chi-square, non-parametrics, multiple regression) are explicitly deferred to the Pyodide kernel (priority: later), gated behind an "Advanced" action.

STACK APPROACH: client-side React component. Computation runs synchronously in-browser for the JS battery (datasets in a thesis are small). No server round-trip for the math -> nothing leaves the researcher's machine except what they choose to save. The result is rendered with recharts (already in deps): scatter+regression line, box plots, bar+error-bar. Money-blind by construction -- the analysis operates only on the columns the researcher provides and on the money-free atlas projection; the firewall CHECK (evidence_json_is_clean pattern) is reused if any output is persisted to the evidence substrate.

OUTPUT: an analysis_run record = { method, input_columns, parameters, dataset_hash, results_json (stat, df, p, CI, effect size), figure_spec, created_at, software_versions }. Rendered as a clean result card (statistic, p-value, CI, interpretation sentence with [EKLE:] placeholders for anything the researcher must confirm -- never auto-asserted as fact, per data-integrity rule) plus the recharts figure. The run is bindable to a chapter (embeds as a figure + an auto-written methods/results stub the researcher edits).

PROVENANCE: every run stores the dataset content-hash + the exact method + parameters + library versions, so the run is REPRODUCIBLE -- re-running the same dataset+method yields the same result, and the record proves it. This is the OSF/Jupyter "one-document" reproducibility lesson, and it is the hook into the receipt: a finished run can be minted into a money-blind Provenance Receipt (object_type='analysis') that is citable + credited + provably untainted -- which Overleaf (no repro), SPSS (no provenance), and Benchling (captive) cannot jointly do.

SCHEMA (new tables, all RLS-mirroring thesis_tracks: student + supervisor only):
- thesis_dataset(id, thesis_id FK, name, source ENUM[upload|atlas_query], columns_json, row_count, content_hash, storage_ref nullable, created_by, created_at)
- thesis_analysis_run(id, thesis_id FK, dataset_id FK, method TEXT, input_columns_json, parameters_json, results_json, figure_spec_json, software_versions_json, dataset_hash, created_by, created_at)  -- append-only; re-runs create new rows (version history for free)
RPCs (SECURITY DEFINER, set search_path=public, grant authenticated): save_thesis_dataset, run_is recorded client-side then persisted via save_thesis_analysis_run, list_thesis_analysis_runs, mint_analysis_receipt (extends get_chain_receipt). 0-ERROR advisors is the bar.

BUILD ORDER inside the tool: (1) dataset import+hash, (2) the five JS tests + result card, (3) recharts figure, (4) bind-to-chapter, (5) mint-receipt. Ship (1)-(3) first as a real used thing before wiring the moat hooks.

## Data gravity
Gravity accrues as INTERLINKED, CREDENTIALED STATE -- the kind you cannot recreate elsewhere without losing what makes it valuable:

1. THE REFERENCE LIBRARY (Zotero gravity): every DOI/BibTeX a researcher imports, and every atlas Provenance Receipt they cite, accumulates in their per-thesis library. Other reference managers cannot resolve a /receipt/[pid] as a money-blind citable source -- so the library is partly composed of references that only exist because GEOCON exists.

2. THE LIVE DOCUMENT (Overleaf gravity): chapters authored against the library, with inline citations and embedded figures. Leaving means re-citing every inline reference by hand and abandoning the live thesis the supervisor is also bound to.

3. THE DATASET + ANALYSIS CHAIN (Benchling gravity, OPEN version): datasets bound to atlas species IDs; analysis runs bound to dataset hashes; figures bound to runs; chapters bound to figures. This typed, interlinked substrate (the same kind as tics+evidence and the 279-node chain) is far harder to leave than loose files because leaving breaks the links -- the dataset loses its species binding, the figure loses its reproducible run, the result loses its provenance.

4. THE REPRODUCIBILITY + TIMESTAMP RECORD (OSF gravity): each analysis run is a timestamped, hash-anchored, re-runnable record. That record's VALUE is that it is locked, dated, and verifiable -- a property that evaporates the instant you copy the numbers into Word.

5. THE CREDIT + CITATION LEDGER (the GEOCON-only gravity): minted receipts surface on the public Researcher Passport, accrue credits, and notify the researcher when their work is cited or enables an outcome. This reputational accrual is the strongest, most defensible gravity because it is external credentialing -- it lives on the platform and travels with the researcher's name, so leaving forfeits accumulated, verifiable credit.

CRUCIAL DISTINCTION: this is OPEN gravity, not captive gravity. Everything exports (BibTeX, CSL-JSON, CSV, LaTeX, DOCX). The researcher stays not because they CAN'T leave but because leaving costs the citations-to-the-atlas, the reproducibility proof, the supervisor's shared state, and the public credit -- none of which any external toolchain can reproduce. That is the firewall-aligned, durable form of data gravity.

- **Collaboration and supervision binding other people to the state** [high] — Use existing Supabase Realtime + RLS + program visibility gating (fn_program_can_see_interior, join requests, owner/assignee roles) for Overleaf-style co-authored theses and Benchling-style inline supervisor comments on drafts. Switching cost rises sharply once a supervisor and co-authors depend on the same live document -- retention via other people, the second-strongest lever after the credential.
- **The gap-as-product engagement loop** [medium] — Surface the open knowledge-gaps (the atlas of what we do not yet know) as a personalized, evidence-shaped worklist on the Workspace: gaps near a researcher's species_set, mission tags, and program, each closable into a citable receipt. This converts the North-Star (move a fact from 0) into a recurring reason to return, and ties re-engagement directly to credentialed output rather than streaks.
- **Reputation / Passport as portable identity that points home** [medium] — Enrich the public Researcher Passport to show thesis works-in-progress and defended theses, receipt-backed contributions, co-author network, and a one-click CV/contribution export -- a persistent link for email signatures and ORCID. The card is portable (you can take it anywhere) but every credential on it resolves back to a GEOCON receipt, so the profile advertises GEOCON while serving the researcher.
- **Honest, event-driven re-engagement notifications** [medium] — Notify only on real, consequential events the researcher would want: your edit was accepted and a receipt minted, a supervisor commented on your draft, your thesis/grant was cited, a gap in your watched species opened or closed, a join request arrived. Tie to the existing notification prefs panel; default to digest, fully user-controlled. Re-engagement that reports genuine state changes, never manufactured urgency.
- **Cross-surface Workspace sync (the unifying to-do)** [low] — Cascade thesis milestones, grant sections, assignments (tics), and open gaps into one Workspace view so the researcher's whole working life has a single home. Breadth where each surface references the others creates gravity no single feature does -- but only after each surface is actually used (avoid the cathedral trap of shipping empty layers).

## The one lock-in
The money-blind, citable Provenance Receipt. It is the one thing no competitor can offer: a persistent-identifier, tamper-evident credential that a researcher's contribution was provably never contaminated by commerce -- enforced as a wall in Postgres, not a promise. A researcher cannot leave because the citations, CV entries, and ORCID links they have built point at receipts that only GEOCON can issue and only GEOCON's structural firewall can vouch for. It is the strongest lock-in precisely because it is reputational and non-coercive: the researcher gains a credential they want to keep, rather than being held hostage by un-exportable data. Overleaf has the document, Zotero has the library, OSF has the timestamp -- only GEOCON binds reproducibility + credit + provable money-blindness into a single citable object.

## Integrity guardrails
Lock-in must come from genuine accumulated value and the researcher's own credential, never from captivity. Concretely: (1) Full data ownership and export at all times -- one-click export of the reference library (BibTeX/CSL-JSON/RIS), the thesis (LaTeX/DOCX/MD), analysis scripts + data, and all receipts as a portable bundle; the Benchling export-captivity model is explicitly rejected as the WRONG moat (a documented user grievance and contradictory to the firewall/portability stance). (2) Receipts and credit are earned only by evidenced, provenance-labelled facts -- never fabricated, never auto-saved from AI output (AI drafts use [EKLE:]/[ADD:] placeholders and require human review). (3) No dark patterns: no fake urgency, no streak guilt, no roach-motel cancellation; notifications report real events and are user-controlled with a digest default. (4) The firewall holds -- zero money columns touch the evidence substrate; commerce may only cite conservation, one-directional, read-only. (5) Credit is attributable and fair: co-authors and supervisors are visibly credited, never erased. The deal offered to the researcher is honest: stay because your work is more citable, more credited, and more trusted here than anywhere else -- and walk out the door any day with everything you made.

## Hardest risks
- Cathedral trap at ~0 users: shipping six empty surfaces (canvas, library, analysis, figures, receipts, passport) none of which is actually used. Mitigation is structural in this plan -- Phase 0 ships ONLY the founder's stats tool, end-to-end and independently useful, before any breadth; each later surface must be used before the next is built.
- Cold-start / network-dependent value: collaboration, co-author networks, citation-of-your-work, and re-engagement notifications have near-zero value at 0 external users. Single-player value (import a dataset, run an analysis, build a library, write a chapter, mint a self-receipt) must carry retention until a network exists -- the founder must be able to use the whole core solo on day one. This is why all such mechanisms are deferred to Phase 3.
- Receipt credibility erosion: if receipts are minted for thin or unverified facts, the credential is devalued and the entire moat weakens. Mitigation: gate minting behind genuine evidence strength; keep the money-blind CHECK and allowlist projection strict; treat the receipt/firewall changes as security-sensitive (run /crosscheck + /security-review per CLAUDE.md).
- Statistical correctness liability: a wrong p-value or mislabeled test in a thesis is a serious failure. Pure-JS libraries (jStat regression is 'not yet included'; chi-square undocumented) have real gaps -- must route only the verified five tests through the JS battery, label every result with method + library version, and defer the long tail to Pyodide/scipy rather than approximating it. Cross-check a sample of outputs against R/scipy before trusting the pane.
- Pyodide weight and memory ceiling (~10MB+ first load, tab crashes on large datasets): real, but contained by keeping pure-JS stats on the critical path and making Pyodide opt-in, Worker-isolated, and chunked (Phase 3 only).
- Integrity drift toward dark patterns under growth pressure (streaks, export friction, manufactured urgency). Mitigation: the open-export ownership guarantee and notification-honesty are stated constraints, cross-checkable against the North-Star and the firewall; the Benchling export-captivity model is explicitly rejected as the wrong moat.
- Scope sprawl into NVivo-class qualitative analysis or full SPSS parity -- high build cost, low fit for a geophyte conservation thesis. Defer indefinitely unless researchers are demonstrably qualitative.

## Open questions
- Audience reality check: is the first real user the founder himself (solo, cold start), and is the founder's own thesis/analysis the day-one test case? That decides whether Phase 0 optimizes for single-player depth over any collaboration affordance.
- Quantitative vs qualitative: are the target geophyte theses primarily empirical/quantitative (germination rates, alkaloid concentration, population morphometrics)? If a meaningful share is qualitative, the pure-JS stats battery is the wrong first bet and the priority should shift.
- Receipt scope decision (your call, firewall-adjacent): should a single analysis run be mintable into a Provenance Receipt, or only a completed/defended chapter? This sets how heavy the verification gate must be to keep the credential credible.
- Citation infrastructure dependency: are we comfortable leaning on CrossRef/DOI + ORCID for the library and Passport, given outages/policy changes? The plan keeps the GEOCON receipt self-sufficient as the primary artifact with ORCID as enrichment -- confirm that ranking.
- Export bundle definition: what exactly is the one-click 'take everything and leave' bundle (library BibTeX/CSL-JSON + thesis LaTeX/DOCX/MD + datasets CSV + analysis scripts + receipts)? Defining this now keeps the open-gravity promise honest as features accrete.
- Pyodide go/no-go: is the long-tail advanced-stats need real enough among target theses to justify the ~10MB+ payload and memory work, or should the system stop at the verified five-test JS battery and tell heavy users to bring results in as a dataset?
- Naming and placement: does this live as a tab inside the existing /geocon/thesis route (lowest-friction, reuses thesis_tracks), or does it deserve a distinct 'Workbench' surface? The plan assumes the former to avoid a new silo -- confirm that framing.
